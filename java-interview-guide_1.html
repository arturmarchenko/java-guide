9 вопросов▼
  • 1Что означает public static void main(String[] args)? Разбери каждое ключевое слово.▶
    public — модификатор доступа, метод видим из любого класса.
    static — принадлежит классу, не объекту. JVM вызывает без создания экземпляра.
    void — ничего не возвращает.
    main — точка входа, JVM ищет именно это имя.
    String[] args — массив аргументов командной строки.
  • 2Назови все 8 примитивов. Чем они отличаются от объектов?▶
    Примитивы: byte, short, int, long, float, double, char, boolean.

    Хранятся на стеке, передаются по значению, нет методов, дешевле по памяти. Объекты — в Heap, передаются по ссылке, могут быть null.
    У каждого примитива есть обёртка: Integer, Long, Double, Boolean и т.д. — нужны для коллекций, дженериков, nullable значений.
  • 3Что такое autoboxing и unboxing? Какие подводные камни?▶
    Autoboxing — автоматическая конвертация примитива в обёртку: Integer i = 5;
    Unboxing — обратное: int x = i;
    Ловушка 1: NullPointerException при unboxing null → Integer i = null; int x = i; — NPE!
    Ловушка 2: Integer кешируется только для -128..127. Integer.valueOf(200) == Integer.valueOf(200) → false! Используй .equals().
    Integer a = 127; Integer b = 127;
System.out.println(a == b); // true (кеш)
Integer c = 128; Integer d = 128;
System.out.println(c == d); // false (разные объекты!)
  • 4Чем отличается final для переменной, метода и класса?▶
    final переменная — нельзя переприсвоить ссылку (но изменить объект — можно!).
    final метод — нельзя переопределить (override) в наследнике.
    final класс — нельзя наследовать. Пример: String, Integer — final классы.
    static final = константа: public static final int MAX_SIZE = 100;
    final поле ссылочного типа: ссылку менять нельзя, но объект можно:
    final List<String> list = new ArrayList<>(); list.add("ok"); // OK
  • 5Что такое static? Когда использовать статические поля и методы?▶
    static — член принадлежит классу, а не экземпляру. Одна копия на все объекты.

    Когда использовать:
    • Утилитные методы без состояния: Math.sqrt(), Collections.sort()
    • Константы: public static final
    • Фабричные методы: LocalDate.of()
    • Singleton паттерн
    • Счётчики экземпляров
    static поля — общее состояние для всех потоков. В многопоточности нужна синхронизация!
  • 6Java передаёт параметры по значению или по ссылке?▶
    Всегда по значению! Для объектов передаётся значение ссылки (копия адреса).
    void modify(int x) { x = 10; } // не изменит оригинал

void modify(List<String> list) {
  list.add("hi");         // изменит объект — та же ссылка
  list = new ArrayList(); // не изменит переменную снаружи
}

int n = 5; modify(n); // n всё ещё 5
List<String> lst = new ArrayList<>();
modify(lst); // lst содержит "hi"
  • 7Что такое varargs (...)? Как работает под капотом?▶
    Varargs — передача произвольного кол-ва аргументов. Компилятор превращает в массив.
    void print(String... args) {
  for (String s : args) System.out.println(s);
}
print("a", "b", "c");              // компилятор: new String[]{"a","b","c"}
print(new String[]{"a", "b"});     // явный массив тоже OK
    Varargs должен быть последним параметром. Нельзя два varargs в одном методе.
  • 8Что нового в Java 8? Главные фичи.▶
    • Lambda expressions — (a, b) -> a + b
    • Stream API — функциональная обработка коллекций
    • Optional<T> — обёртка для nullable значений
    • default/static методы в интерфейсах
    • Method references — String::toLowerCase, User::new
    • Новый Date/Time API — LocalDate, LocalDateTime, ZonedDateTime
    • CompletableFuture — неблокирующая асинхронность
  • 9Что такое Optional? Как правильно использовать?▶
    Optional — контейнер, явно указывающий что значение может отсутствовать. Исключает NPE при правильном использовании.
    Optional<User> user = repo.findById(id);
user.ifPresent(u -> System.out.println(u.getName()));
String name = user.map(User::getName).orElse("Unknown");
User u = user.orElseThrow(() -> new NotFoundException(id));
    Антипаттерны: opt.get() без isPresent(), Optional как поле класса, Optional как параметр метода. Optional предназначен для возвращаемых значений!

8 вопросов▼
  • 1Иерархия исключений в Java. Checked vs Unchecked vs Error.▶
    Throwable
├── Error (OutOfMemoryError, StackOverflowError) — не ловить!
└── Exception
    ├── Checked (IOException, SQLException) — компилятор требует обработки
    └── RuntimeException (Unchecked) — компилятор не требует
        ├── NullPointerException
        ├── IllegalArgumentException
        ├── ClassCastException
        └── ...
    Checked — для восстанавливаемых ситуаций (файл не найден, нет сети). Unchecked — для ошибок программирования.
  • 2Разница между throw и throws?▶
    throw — выброс конкретного исключения прямо сейчас:
    throw new IllegalArgumentException("age must be positive");

    throws — объявление в сигнатуре что метод может выбросить:
    void readFile(String path) throws IOException, FileNotFoundException {}
    throws нужен только для Checked исключений — для RuntimeException необязателен.
  • 3Что такое try-with-resources? Чем лучше finally?▶
    Автоматически закрывает ресурсы (AutoCloseable) после блока. Несколько ресурсов — закрываются в обратном порядке.
    try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
  // используем
} // ps.close(), затем conn.close() — автоматически
    Преимущество перед finally: если исключение в close() — оно становится suppressed, оригинальное не теряется. С finally — оригинальное исключение терялось!
  • 4Выполнится ли finally, если вызвать System.exit() или если поток убит?▶
    System.exit() — finally НЕ выполнится. JVM завершается немедленно.
    Thread.stop() (deprecated) — finally может не выполниться.
    Thread.interrupt() — finally выполнится.
    В остальных случаях — finally всегда выполняется, даже при return в try.
    Порядок: try → если исключение → catch → finally → (продолжение или re-throw)
  • 5Что такое multi-catch (Java 7)?▶
    // До Java 7 — дублирование кода
try { risky(); }
catch (IOException e) { handle(e); }
catch (SQLException e) { handle(e); }

// Java 7+ — multi-catch
try { risky(); }
catch (IOException | SQLException e) {
  handle(e); // e — effectively final, нельзя переприсвоить
}
  • 6Как создать своё исключение? Что обязательно сделать?▶
    public class OrderNotFoundException extends RuntimeException {
  private final Long orderId;

  public OrderNotFoundException(Long id) {
    super("Order not found: " + id);
    this.orderId = id;
  }
  // Конструктор с cause — ВАЖЕН для сохранения стектрейса!
  public OrderNotFoundException(Long id, Throwable cause) {
    super("Order not found: " + id, cause);
    this.orderId = id;
  }
  public Long getOrderId() { return orderId; }
}
    Всегда добавляй конструктор с Throwable cause — иначе при оборачивании исключения потеряешь оригинальный стектрейс!
  • 7Что такое Exception chaining (цепочки исключений)?▶
    Оборачивание низкоуровневого исключения в высокоуровневое с сохранением причины.
    try {
  db.save(entity);
} catch (SQLException e) {
  // НЕ ПРАВИЛЬНО: throw new ServiceException("save failed"); — теряем cause
  // ПРАВИЛЬНО:
  throw new ServiceException("Failed to save order", e); // e = cause
}
// В стектрейсе увидим: ServiceException caused by: SQLException
  • 8Можно ли перехватить Error? Когда это уместно?▶
    Технически можно: catch (Error e) или catch (Throwable t).
    В продакшне — плохая практика. Error означает JVM в нестабильном состоянии. Единственные исключения:
    - Логирование с последующим re-throw в глобальных обработчиках
    - AssertionError в тестах
    - StackOverflowError для защиты рекурсии (редко)

String, StringBuilder, StringBuffer — всё про строки
immutability, String Pool, intern(), методы, производительность
7 вопросов▼
  • 1Почему String immutable? 4 причины.▶
    1. String Pool — JVM кеширует литералы. Если бы String был изменяемым, один поток изменил бы строку другого через пул.
    2. Thread-safe — неизменяемые объекты безопасны без синхронизации.
    3. Безопасность — строки используются как пути, пароли, сетевые адреса. Изменяемость = уязвимость.
    4. hashCode кеш — вычисляется один раз при первом вызове, затем кешируется. Ускоряет HashMap.
  • 2Что такое String Pool? Как работает intern()?▶
    String Pool (String Intern Pool) — специальная область в Heap для уникальных строковых литералов.
    String a = "hello";             // в пул
String b = "hello";             // тот же объект из пула
String c = new String("hello"); // НОВЫЙ объект в Heap, минуя пул
System.out.println(a == b);     // true
System.out.println(a == c);     // false
System.out.println(a == c.intern()); // true — intern() помещает в пул
    intern() может экономить память при большом числе одинаковых строк, но вызывает GC pressure на пул. Используй осознанно.
  • 3String vs StringBuilder vs StringBuffer — когда что?▶
    String — immutable. Каждая операция создаёт новый объект. Для констант и редких операций.
    StringBuilder — mutable, НЕ thread-safe. Для конкатенации в одном потоке. Быстрый.
    StringBuffer — mutable, thread-safe (все методы synchronized). Медленнее. Почти не нужен — в многопоточности обычно нет смысла строить строку одновременно.
    Компилятор автоматически заменяет + на StringBuilder, но только вне циклов. В цикле — явный StringBuilder.
  • 4Что происходит при конкатенации строк в цикле через +?▶
    // ПЛОХО — O(n²), каждый шаг создаёт новую строку
String result = "";
for (int i = 0; i < 10000; i++) {
  result += i; // новый объект String каждый раз!
}

// ХОРОШО — O(n), один объект
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
  sb.append(i);
}
String result = sb.toString();
  • 5Чем == отличается от .equals() для строк?▶
    == — сравнивает ссылки (адреса в памяти).
    .equals() — сравнивает содержимое.
    НИКОГДА не сравнивай строки через ==. Используй:
    str.equals("hello") — если str может быть null → NPE!
    "hello".equals(str) — безопаснее
    Objects.equals(str1, str2) — null-safe, лучший вариант
  • 6Важнейшие методы String — что точно нужно знать?▶
    equals(), equalsIgnoreCase() — сравнение
    contains(), startsWith(), endsWith() — поиск
    indexOf(), lastIndexOf() — позиция
    substring(from, to) — подстрока
    split(regex, limit) — разделение
    trim() — обрезка ASCII пробелов, strip() — Unicode-aware (Java 11+)
    replace(old, new), replaceAll(regex, replacement)
    String.join(delim, elements) — объединение
    String.format(template, args) — форматирование
    toCharArray(), chars() — работа с символами
    matches(regex) — проверка по паттерну
  • 7Что такое Text Blocks (Java 15+)?▶
    Многострочные строки без экранирования:
    // Старый способ
String json = "{\n  \"name\": \"John\",\n  \"age\": 30\n}";

// Text Block (Java 15+)
String json = """
    {
      "name": "John",
      "age": 30
    }
    """;
// Отступы автоматически убираются относительно закрывающего """
    Идеально для SQL, JSON, HTML в коде. Читаемость — на порядок лучше.

10 вопросов▼
  • 1Что такое инкапсуляция? Зачем нужны геттеры и сеттеры?▶
    Инкапсуляция — сокрытие внутреннего состояния, предоставление доступа только через методы.
    public class BankAccount {
  private double balance; // скрытое состояние

  public void deposit(double amount) {
    if (amount <= 0) throw new IllegalArgumentException();
    this.balance += amount; // валидация + бизнес-логика
  }
  public double getBalance() { return balance; } // только чтение
  // нет setBalance() — нельзя поставить произвольный баланс!
}
    Можно изменить внутреннюю реализацию (например, хранить в копейках) не меняя публичный API. Контроль инвариантов объекта.
  • 2Что такое наследование? Composition over Inheritance — почему?▶
    Наследование — получение полей и методов родителя через extends. В Java одиночное.

    Проблемы наследования:
    • Хрупкий базовый класс — изменение родителя ломает всех наследников
    • Нарушение LSP — наследник может вести себя неожиданно
    • Тесная связанность, сложно тестировать
    • Невозможно изменить решение без рефакторинга
    // Наследование — тесная связь
class Dog extends Animal { ... }

// Композиция — гибко
class Dog {
  private final Behavior behavior; // можно подменить
  private final Sound sound;
}
    «Prefer composition over inheritance» — Effective Java, Item 18. Наследование уместно когда есть чёткое отношение IS-A.
  • 3Compile-time vs Runtime полиморфизм — объясни с примерами.▶
    Compile-time (статический) — overloading:
    class Printer {
  void print(int x)    { System.out.println("int: " + x); }
  void print(String s) { System.out.println("str: " + s); }
  // Компилятор выбирает метод по типу аргумента
    Runtime (динамический) — overriding:
    Animal a = new Dog(); // Dog переопределяет speak()
a.speak(); // JVM смотрит реальный тип (Dog) и вызывает Dog.speak()
// Это называется virtual method invocation / dynamic dispatch
  • 4Overloading vs Overriding — ключевые отличия. Что нельзя переопределить?▶
    Overloading: тот же класс, то же имя метода, разные параметры. Compile-time. Возвращаемый тип может отличаться.
    Overriding: наследник меняет реализацию метода родителя. Runtime. Сигнатура должна совпадать. Аннотация @Override — защита от опечаток.

    Нельзя переопределить:
    • static методы — можно скрыть (hiding), но не override
    • final методы
    • private методы
    • Конструкторы (они не наследуются)
  • 5Abstract class vs Interface — когда что выбрать? Изменения в Java 8.▶
    Abstract class: может иметь состояние (поля), конструкторы, реализованные методы. Один extends.
    Interface: контракт. До Java 8 — только абстрактные методы. Java 8+: default и static методы. Java 9+: private методы. Несколько implements.

    Когда abstract class: Template Method паттерн, общее состояние/поведение, частичная реализация.
    Когда interface: контракт для DI, multiple "types" для одного класса, API.
    // Template Method с abstract class
abstract class Report {
  final void generate() { // final — нельзя переопределить шаблон
    fetchData();
    String content = formatData(); // abstract
    save(content);
  }
  abstract String formatData();
  void fetchData() { /* дефолтная реализация */ }
}
  • 6Зачем default методы в интерфейсах? Проблема diamond?▶
    Добавить новые методы в интерфейс без поломки существующих реализаций. Пример: forEach() в Iterable — тысячи классов получили его автоматически.
    interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }

class C implements A, B {
  // Компилятор требует явного переопределения при конфликте!
  @Override
  public void hello() { A.super.hello(); } // явный выбор
}
  • 7SOLID принципы — объясни каждый с примером нарушения.▶
    S — Single Responsibility: класс имеет одну причину для изменения.
    Нарушение: UserService и сохраняет в БД, и отправляет email, и генерирует PDF.

    O — Open/Closed: открыт для расширения, закрыт для изменения.
    Нарушение: switch/if по типам — новый тип = правка существующего кода.

    L — Liskov Substitution: наследник заменяет родителя без сюрпризов.
    Нарушение: Square extends Rectangle нарушает LSP — изменение ширины меняет высоту.

    I — Interface Segregation: много маленьких интерфейсов лучше одного большого.
    Нарушение: интерфейс Worker с методами work() и eat() — роботы не едят!

    D — Dependency Inversion: зависи от абстракций.
    Нарушение: OrderService напрямую создаёт new MySQLOrderRepo().
  • 8Что произойдёт если переопределить equals() но не hashCode()?▶
    Нарушение контракта: «если a.equals(b) == true, то a.hashCode() == b.hashCode()».
    class User {
  String name;
  @Override
  public boolean equals(Object o) {
    return o instanceof User u && name.equals(u.name);
  }
  // hashCode НЕ переопределён → Object.hashCode() = адрес памяти
}
Map<User, String> map = new HashMap<>();
User u1 = new User("John"); // hashCode = 12345
User u2 = new User("John"); // hashCode = 67890 (другой!)
map.put(u1, "admin");
map.get(u2); // null! Ищет в корзине 67890, а u1 в корзине 12345
new HashSet<>(List.of(u1, u2)).size(); // 2, а не 1!
  • 9Как создать immutable класс? Правила.▶
    public final class Money { // 1. final класс
  private final long amount;      // 2. final поля
  private final Currency currency; // 3. final поля

  public Money(long amount, Currency currency) {
    this.amount = Objects.requireNonNull(amount);
    this.currency = Objects.requireNonNull(currency);
  }
  public long getAmount() { return amount; }    // 4. только геттеры
  public Currency getCurrency() { return currency; }

  public Money add(Money other) { // 5. операции возвращают новый объект
    if (!currency.equals(other.currency)) throw new IllegalArgumentException();
    return new Money(amount + other.amount, currency);
  }
  // 6. если поле — mutable объект (List, Date), возвращать defensive copy
}
  • 10Что такое instanceof и pattern matching (Java 16+)?▶
    // Старый способ
if (obj instanceof String) {
  String s = (String) obj; // явный cast
  System.out.println(s.length());
}

// Java 16+ — pattern matching
if (obj instanceof String s) { // bind variable сразу!
  System.out.println(s.length()); // s уже типа String
}

// Java 21 — switch pattern matching
String result = switch (obj) {
  case Integer i -> "int: " + i;
  case String s when s.length() > 5 -> "long string: " + s;
  case null -> "null";
  default -> "other";
};

10 вопросов▼
  • 1Пошагово: что происходит при map.put(key, value)?▶
    1. Вычисляется key.hashCode().
    2. Перемешивание хеша: hash = h ^ (h >>> 16) — снижает коллизии при малой capacity.
    3. Индекс корзины: index = hash & (capacity - 1) (аналог % но быстрее).
    4. Если корзина пустая → кладём Node{hash, key, value, next=null}.
    5. Если занята (коллизия) → проходим по списку/дереву, сравниваем через equals.
    - Ключ найден → перезаписываем value.
    - Не найден → добавляем в конец.
    6. Если size > capacity * loadFactor → resize (удвоение).
    Начальная capacity = 16. LoadFactor = 0.75. Resize при 12 элементах.
  • 2Что такое коллизия? Как решается в Java 8+?▶
    Коллизия — два разных ключа дают одинаковый индекс корзины.

    До Java 8: цепочки (chaining) — LinkedList в корзине. Поиск O(n) при массе коллизий.
    Java 8+: если в корзине ≥ 8 элементов И capacity ≥ 64 → treeify: список → красно-чёрное дерево. Поиск O(log n).
    Когда корзина уменьшается до 6 → обратно в список (untreeify).
    Намеренные коллизии (все ключи с одинаковым hashCode) — атака типа HashDoS. Именно поэтому в Java 8 добавили treeify.
  • 3Что такое resize? К чему ведёт в многопоточности?▶
    Resize — создание нового массива capacity×2 и перехеширование всех элементов. O(n).
    Элементы распределяются в new_index = old_index или old_index + oldCapacity.
    Java 7: два потока делающих resize одновременно → бесконечный цикл в linked list → 100% CPU вечно! В Java 8 это исправлено, но race condition всё равно приводит к потере данных.
    Вывод: используй ConcurrentHashMap в многопоточности.
    Оптимизация: new HashMap<>(expectedSize * 4 / 3 + 1) — избежать resize.
  • 4Можно ли использовать null в HashMap? А в других Map?▶
    HashMap — один null-ключ (в корзине [0]), любое число null-значений. ✓
    LinkedHashMap, TreeMap — null-значения OK; null-ключ: TreeMap → NPE при сравнении. ✗
    Hashtable — null ключ или значение → NPE. ✗
    ConcurrentHashMap — null ключ или значение → NPE. ✗
    ConcurrentHashMap запрещает null намеренно: невозможно различить «ключ отсутствует» и «значение равно null» без дополнительных блокировок.
  • 5HashMap vs LinkedHashMap vs TreeMap — разница?▶
    HashMap — нет гарантий порядка. O(1) get/put/remove.
    LinkedHashMap — сохраняет порядок вставки (или порядок доступа — для LRU кеша). Доп. prev/next ссылки. O(1).
    TreeMap — отсортированные ключи (Comparable или Comparator). O(log n). Реализует NavigableMap: floorKey, ceilingKey, subMap, headMap, tailMap.
    // LRU кеш через LinkedHashMap
new LinkedHashMap<>(capacity, 0.75f, true) {
  @Override protected boolean removeEldestEntry(Map.Entry eldest) {
    return size() > capacity;
  }
};
  • 6Что будет если у всех ключей hashCode() возвращает константу?▶
    Все элементы попадут в одну корзину. HashMap превращается в LinkedList (или дерево при ≥8).
    get/put → O(n) вместо O(1). При 10000 элементах — 10000 вызовов equals при каждом запросе.
    Классический вопрос-ловушка. HashMap будет работать корректно, но крайне медленно. Именно поэтому важно хороший hashCode.
  • 7Что такое fail-fast и fail-safe итераторы? ConcurrentModificationException?▶
    Fail-fast — при изменении коллекции во время итерации бросает ConcurrentModificationException. Реализован через счётчик modCount: итератор запоминает modCount при создании, проверяет при каждом next().
    Пример: ArrayList, HashMap, HashSet.

    Fail-safe — итерирует по снимку (snapshot), исключений нет, данные могут быть устаревшими.
    Пример: ConcurrentHashMap, CopyOnWriteArrayList.
    // Как правильно удалять при итерации:
Iterator<String> it = list.iterator();
while (it.hasNext()) {
  if (condition(it.next())) it.remove(); // через итератор — OK
}
// Java 8+:
list.removeIf(s -> condition(s)); // removeIf — безопасно
  • 8computeIfAbsent vs putIfAbsent vs getOrDefault?▶
    getOrDefault(key, def) — вернуть значение или дефолт, не изменяет Map.
    putIfAbsent(key, value) — добавить если нет. Значение создаётся ВСЕГДА, даже если ключ есть.
    computeIfAbsent(key, fn) — функция вызывается только если ключа нет. Ленивая инициализация.
    // Группировка — классический паттерн
Map<String, List<User>> byCity = new HashMap<>();
users.forEach(u ->
  byCity.computeIfAbsent(u.getCity(), k -> new ArrayList<>()).add(u)
);
  • 9WeakHashMap, IdentityHashMap, EnumMap — когда использовать?▶
    WeakHashMap — ключи через WeakReference. GC удаляет запись когда ключ недостижим. Для кешей, ClassLoader кешей.

    IdentityHashMap — сравнивает ключи через == (ссылки), а не equals. Для сериализации, определения циклов в графах.

    EnumMap — ключи-enum. Внутри простой массив по ordinal(). O(1), меньше памяти чем HashMap.
    Всегда используй EnumMap вместо HashMap когда ключи — enum значения. Аналогично EnumSet вместо HashSet.
  • 10Как работает Stream API? Что такое lazy evaluation?▶
    Stream — конвейер операций. Промежуточные (filter, map, sorted) — ленивые, не выполняются. Терминальная (collect, forEach, count) — запускает весь конвейер.
    // Без терминальной — НИЧЕГО не выполнится
Stream<User> stream = users.stream()
  .filter(u -> { System.out.println("filter"); return u.isActive(); });
// Тут ни одного "filter" ещё!
stream.collect(Collectors.toList()); // теперь выполняется

// Short-circuit: findFirst останавливает обработку как только найдёт
Optional<User> first = users.stream()
  .filter(u -> u.getAge() > 18)
  .findFirst(); // не обрабатывает весь список!
    parallelStream() — ForkJoinPool. Эффективен при CPU-bound операциях и больших коллекциях. При малых коллекциях overhead от параллелизма хуже.

  • 1ArrayList vs LinkedList — сложности и когда что выбрать?▶
    Операция         ArrayList    LinkedList
get(i)           O(1)         O(n)
add(конец)       O(1)*        O(1)
add(i)           O(n)         O(n) [O(1) если есть итератор]
remove(i)        O(n)         O(n) [O(1) если есть итератор]
contains         O(n)         O(n)
* амортизированно
    На практике ArrayList почти всегда быстрее из-за cache locality процессора. Элементы ArrayList в памяти рядом → меньше cache miss. LinkedList — каждый узел отдельный объект в куче.
    LinkedList реально выигрывает только при очень частых вставках/удалениях В СЕРЕДИНУ с уже известным итератором. Для дек/стека используй ArrayDeque.
  • 2Как устроен ArrayList? Что происходит при переполнении?▶
    ArrayList хранит элементы в Object[] elementData. Начальная capacity = 10 (или 0 при new ArrayList<>() без size).

    При переполнении (size == capacity):
    1. newCapacity = oldCapacity + (oldCapacity >> 1) = capacity × 1.5
    2. Arrays.copyOf(elementData, newCapacity) — O(n)
    3. Старый массив становится мусором для GC.
    Оптимизация: знаешь размер → new ArrayList<>(expectedSize) или ensureCapacity(). Избегает повторных расширений.
  • 3HashSet vs TreeSet vs LinkedHashSet — разница?▶
    HashSet — обёртка над HashMap (value = константный объект). Нет порядка. O(1) add/remove/contains.
    TreeSet — обёртка над TreeMap. Отсортированный порядок. O(log n). Методы: first(), last(), headSet(), tailSet(), subSet().
    LinkedHashSet — обёртка над LinkedHashMap. Порядок вставки. O(1). Чуть больше памяти.
  • 4Comparable vs Comparator — разница и применение?▶
    Comparable — объект сам определяет «естественный» порядок через compareTo(). Один порядок на класс.
    Comparator — внешняя стратегия сравнения. Множество разных порядков.
    // Comparable
class User implements Comparable<User> {
  @Override public int compareTo(User other) {
    return this.name.compareTo(other.name);
  }
}
// Comparator — Java 8+
Comparator<User> comp = Comparator
  .comparing(User::getAge)
  .thenComparing(User::getName)
  .reversed();
users.sort(comp);
  • 5Что такое PriorityQueue? Как работает мин-куча?▶
    PriorityQueue — очередь с приоритетами. Внутри — бинарная мин-куча (min-heap) на массиве.
    poll() всегда возвращает минимальный элемент. O(log n) для add()/poll(). O(1) для peek().
    PriorityQueue<Integer> minHeap = new PriorityQueue<>();
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

// Топ-K наименьших из потока:
PriorityQueue<Integer> topK = new PriorityQueue<>(Comparator.reverseOrder());
for (int n : numbers) {
  topK.offer(n);
  if (topK.size() > k) topK.poll(); // удаляем максимум
}
    PriorityQueue НЕ гарантирует порядок при итерации — только poll() возвращает в правильном порядке.
  • 6ArrayDeque vs LinkedList для стека и очереди?▶
    Оба реализуют Deque (двусторонняя очередь), но ArrayDeque быстрее.
    Deque<String> stack = new ArrayDeque<>(); // стек
stack.push("a"); stack.push("b");
stack.pop(); // "b"

Deque<String> queue = new ArrayDeque<>(); // очередь
queue.offer("a"); queue.offer("b");
queue.poll(); // "a"
    ArrayDeque — внутри циклический массив, cache-friendly. Быстрее LinkedList и даже Stack (который synchronized и устарел). Используй ArrayDeque для стека и очереди.
  • 7Что такое BlockingQueue? Реализации и применение.▶
    BlockingQueue — очередь с блокирующими операциями: take() ждёт если пусто, put() ждёт если полна.
    • ArrayBlockingQueue(capacity) — ограниченный размер, массив, один lock
    • LinkedBlockingQueue — опционально ограниченный, два lock'а (head + tail)
    • PriorityBlockingQueue — с приоритетами, неограниченный
    • SynchronousQueue — нет буфера, прямая передача между потоками
    • DelayQueue — элементы становятся доступны через заданное время
    ThreadPoolExecutor использует BlockingQueue внутри для хранения задач!

5 вопросов▼
  • 1ConcurrentHashMap vs Hashtable vs synchronizedMap — детальное сравнение?▶
    Hashtable — все публичные методы synchronized на весь объект. Только один поток в любой момент. Устарело.

    Collections.synchronizedMap() — обёртка, тот же эффект: блокируется весь объект. Итерация требует внешней синхронизации.

    ConcurrentHashMap — Java 7: 16 сегментов, блокировка на сегмент. Java 8+: CAS + synchronized только на уровне корзины. Множество потоков работают одновременно с разными корзинами.
    ConcurrentHashMap в 10-100 раз быстрее Hashtable при конкурентном доступе. Всегда используй ConcurrentHashMap.
    ConcurrentHashMap не гарантирует консистентность составных операций: check-then-act всё равно нужно делать через атомарные методы (putIfAbsent, computeIfAbsent).
  • 2Что такое CopyOnWriteArrayList? Когда использовать?▶
    При каждой записи (add, remove, set) создаётся полная копия массива. Чтения — без блокировок, на неизменяемом снимке.
    Когда: чтений намного больше записей — списки слушателей/наблюдателей (EventListeners), кеши конфигурации, whitelist/blacklist правила.
    При частых записях — катастрофа: O(n) при каждом изменении. Итератор видит снимок на момент создания — не видит последующих изменений.
  • 3Можно ли изменять ConcurrentHashMap во время итерации?▶
    Да! ConcurrentHashMap — fail-safe итератор. Не бросает ConcurrentModificationException.

    Итератор weakly consistent: отражает состояние на момент создания, может видеть или не видеть последующие изменения. Нет гарантий полной консистентности.
    Это именно то что нужно в большинстве concurrent сценариев — не упасть с исключением важнее чем видеть все изменения.
  • 4Что такое ConcurrentSkipListMap и когда использовать?▶
    ConcurrentSkipListMap — concurrent аналог TreeMap. Сортированные ключи + thread-safe без блокировок (lock-free через CAS).

    O(log n) для get/put/remove. Методы headMap(), tailMap(), subMap() тоже thread-safe.
    Используй когда нужна отсортированная concurrent Map. ConcurrentHashMap не сохраняет порядок ключей.
  • 5List.of() vs Collections.unmodifiableList() — в чём разница?▶
    Collections.unmodifiableList(list) — обёртка, запрещает изменения через неё. Но исходный list меняется → обёртка тоже меняется.

    List.of() (Java 9+) — полностью неизменяемый. Не допускает null. Компактное представление (нет оверхеда массива Object[]).
    List.copyOf(list) (Java 10+) — неизменяемая копия.
    unmodifiableList не является truly immutable! Если нужна настоящая иммутабельность → List.of() или List.copyOf().

  • 2Состояния потока — все 6 состояний и переходы между ними.▶
    NEW ──start()──▶ RUNNABLE
                    │
              ┌─────┼──────────┐
              ↓     ↓          ↓
           BLOCKED WAITING  TIMED_WAITING
              │     │          │

  • 3Что такое synchronized? На каком уровне работает?▶
    Захватывает intrinsic lock (монитор) объекта. Только один поток может держать монитор одного объекта.
    // Метод экземпляра → монитор this
public synchronized void method() { ... }

// Статический метод → монитор Class объекта
public static synchronized void staticMethod() { ... }

// Блок → монитор явного объекта (гибче!)
synchronized(lock) { ... }
synchronized(this) { ... }
synchronized(MyClass.class) { ... }
    synchronized — reentrant: один поток может войти в несколько synchronized блоков на одном мониторе без deadlock.
    synchronized на методе = synchronized(this) = блокирует ВЕСЬ объект. Лучше использовать блоки с отдельным объектом-локом для лучшей гранулярности.
  • 4Что такое deadlock? Условия возникновения и способы избежать.▶
    4 условия Coffman для deadlock: взаимоисключение, удержание и ожидание, нет принудительного изъятия, циклическое ожидание.
    // Классический deadlock
Thread A: synchronized(lock1) { synchronized(lock2) { ... } }
Thread B: synchronized(lock2) { synchronized(lock1) { ... } }
// A держит lock1, ждёт lock2; B держит lock2, ждёт lock1
    Как избежать:
    • Захватывать локи в одном порядке (нарушает циклическое ожидание)
    • tryLock(timeout) — с таймаутом, при неудаче отпускать всё
    • Минимизировать число одновременных локов
    • Использовать высокоуровневые конструкции (ConcurrentHashMap вместо synchronized Map)
    Диагностика: jstack <pid> или VisualVM показывает потоки в BLOCKED + «Found one Java-level deadlock».
  • 5Что такое race condition? Пример и решения.▶
    Race condition — результат зависит от порядка выполнения потоков.
    // i++ — НЕ атомарна! Три операции: read, increment, write
// Thread 1 читает i=5, Thread 2 читает i=5
// Оба записывают 6 → результат 6 вместо 7!
private int counter = 0;
public void increment() { counter++; } // UNSAFE!
    Решения:
    • AtomicInteger — CAS, lock-free
    • synchronized — простой но медленнее
    • LongAdder — быстрее AtomicLong при высокой конкурентности (Java 8)
    private final AtomicInteger counter = new AtomicInteger(0);
public void increment() { counter.incrementAndGet(); }
  • 6wait(), notify(), notifyAll() — правила использования.▶
    Вызывать только внутри synchronized блока на том же мониторе!
    // Паттерн Producer-Consumer
// Consumer:
synchronized(buffer) {
  while (buffer.isEmpty()) { // WHILE, не if! (spurious wakeups!)
    buffer.wait();           // отпускает монитор и ждёт
  }
  process(buffer.take());
}
// Producer:
synchronized(buffer) {
  buffer.add(item);
  buffer.notifyAll(); // notifyAll безопаснее notify
}
    Всегда используй while, не if: spurious wakeups — поток может проснуться без notify(). notifyAll() безопаснее notify() — notify() будит только один случайный поток.
  • 7volatile — что гарантирует и что нет? Happens-before.▶
    Гарантирует:
    • Видимость: чтение из main memory, не из CPU кеша (L1/L2)
    • Happens-before: запись volatile → видна всем последующим чтениям
    • Запрет переупорядочивания (instruction reordering) вокруг volatile операций
    НЕ гарантирует атомарность составных операций:
    volatile int i; i++; — три операции: read, +1, write — всё ещё race condition!
    // Корректные применения:
volatile boolean shutdown = false; // флаг остановки
volatile Config config = loadConfig(); // одиночная публикация объекта

// Double-checked locking (Java 5+):
private volatile Singleton instance;
if (instance == null) {
  synchronized(this) {
    if (instance == null) instance = new Singleton();
  }
}
  • 8ReentrantLock vs synchronized — когда что выбрать?▶
    ReentrantLock дополнительно умеет:
    • tryLock() — попытка захватить без ожидания
    • tryLock(time, unit) — с таймаутом
    • lockInterruptibly() — прерываемое ожидание
    • Несколько условий: Condition cond = lock.newCondition()
    • Честная блокировка: new ReentrantLock(true)
    • Информация: getQueueLength(), isLocked()
    ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
  // критическая секция
} finally {
  lock.unlock(); // ОБЯЗАТЕЛЬНО в finally!
}
    synchronized автоматически снимает лок при выходе из блока. ReentrantLock — нет! Всегда unlock() в finally.
  • 9ReadWriteLock и StampedLock — когда применять?▶
    ReentrantReadWriteLock — много читателей или один писатель.
    ReadWriteLock rwLock = new ReentrantReadWriteLock();
// Много потоков могут читать одновременно
rwLock.readLock().lock();
try { return data; } finally { rwLock.readLock().unlock(); }
// Только один поток пишет
rwLock.writeLock().lock();
try { data = newData; } finally { rwLock.writeLock().unlock(); }
    StampedLock (Java 8+) — добавляет optimistic read без блокировки:
    StampedLock sl = new StampedLock();
long stamp = sl.tryOptimisticRead(); // без блокировки!
double x = this.x, y = this.y;
if (!sl.validate(stamp)) { // был ли write?
  stamp = sl.readLock(); // тогда полный read lock
  try { x = this.x; y = this.y; } finally { sl.unlockRead(stamp); }
}
  • 10Что такое livelock и starvation? Чем отличаются от deadlock?▶
    Deadlock — потоки заблокированы навсегда, ничего не делают.

    Livelock — потоки активны, не заблокированы, но не могут продвинуться (постоянно «уступают» друг другу). Как два человека в коридоре, пытающиеся разойтись.

    Starvation (голодание) — один поток не получает ресурсов из-за того что другие (приоритетные) постоянно их захватывают. Fair lock решает проблему.
    В реальных системах starvation часто незаметна — поток работает, но очень медленно из-за постоянных ожиданий.

    • Очередь задач (буферизация пиков)

  • 2Виды пулов из Executors factory — когда что использовать?▶
    newFixedThreadPool(n) — фиксированный. CPU-bound: n = кол-во ядер. IO-bound: n = ядра × (1 + waitTime/cpuTime).
    newCachedThreadPool() — растёт по нужде, кешируют 60 сек. IO-bound.
    newSingleThreadExecutor() — один поток, задачи последовательно. Удобно для упорядочивания.
    newScheduledThreadPool(n) — по расписанию.
    newVirtualThreadPerTaskExecutor() — Java 21, virtual threads (Project Loom).
    В продакшне создавай ThreadPoolExecutor напрямую! newCachedThreadPool может создать неограниченно много потоков. newFixedThreadPool использует unbounded queue — может накопить миллионы задач в памяти!
    ExecutorService pool = new ThreadPoolExecutor(
  corePoolSize, maxPoolSize, keepAlive, TimeUnit.SECONDS,
  new ArrayBlockingQueue<>(queueCapacity), // bounded!
  new ThreadPoolExecutor.CallerRunsPolicy() // backpressure
);
  • 3execute() vs submit() — разница?▶
    execute(Runnable) — запустить и забыть. Нет результата. Исключение убивает поток (UncaughtExceptionHandler).
    submit(Runnable/Callable) — возвращает Future<T>. Исключение хранится в Future, бросается при get().
    // execute — исключение теряется если нет UncaughtExceptionHandler
executor.execute(() -> { throw new RuntimeException(); }); // тихо падает

// submit — исключение хранится в Future
Future<?> f = executor.submit(() -> { throw new RuntimeException(); });
try { f.get(); }
catch (ExecutionException e) { e.getCause(); } // вот оно!
  • 4shutdown() vs shutdownNow() — как правильно остановить пул?▶
    shutdown() — мягкая: не принимает новые задачи, ждёт завершения текущих.
    shutdownNow() — жёсткая: interrupt() всем потокам, возвращает List незапущенных задач.
    executor.shutdown();
try {
  // Ждём 60 секунд
  if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
    List<Runnable> pending = executor.shutdownNow();
    log.warn("{} tasks were not executed", pending.size());
    // Ждём ещё раз после shutdownNow
    if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
      log.error("Pool did not terminate");
    }
  }
} catch (InterruptedException e) {
  executor.shutdownNow();
  Thread.currentThread().interrupt(); // восстанавливаем флаг!
}

  • 6ThreadLocal — как работает и опасность утечек памяти?▶
    У каждого потока своё значение ThreadLocal. Thread хранит ThreadLocalMap внутри себя.

  private static final ThreadLocal<String> USER_ID = new ThreadLocal<>();
  public static void set(String id) { USER_ID.set(id); }
  public static String get() { return USER_ID.get(); }
  public static void clear() { USER_ID.remove(); } // ВАЖНО!
}

    Утечка памяти: в пулах потоков поток живёт вечно → ThreadLocal живёт вечно → объекты не собираются GC. Всегда ThreadLocal.remove() в finally!
  • 7AtomicInteger и CAS — как работает под капотом?▶
    CAS (Compare-And-Swap) — атомарная инструкция процессора: «если текущее == ожидаемое → заменить на новое».
    AtomicInteger ai = new AtomicInteger(5);
ai.incrementAndGet();           // CAS loop: while(!cas(old, old+1))
ai.compareAndSet(5, 10);        // если 5 → поставить 10
ai.getAndUpdate(x -> x * 2);   // любое атомарное преобразование
ai.accumulateAndGet(3, Integer::sum); // 5 + 3 = 8
    При высокой конкурентности на одном значении — CAS loop (spin) тратит CPU. LongAdder распределяет обновления по ячейкам → меньше contention → быстрее при высокой нагрузке.
  • 8CountDownLatch, CyclicBarrier, Semaphore, Phaser — что для чего?▶
    CountDownLatch — ждать пока N событий не произойдут. Одноразовый.
    CountDownLatch latch = new CountDownLatch(3);
// 3 worker потока вызывают latch.countDown()
latch.await(); // главный поток ждёт трёх
    CyclicBarrier — N потоков ждут друг друга в точке. Многоразовый.
    Semaphore — ограничение числа одновременных обращений (пул соединений, rate limiting).
    Semaphore sem = new Semaphore(10); // max 10 одновременно
sem.acquire(); try { doWork(); } finally { sem.release(); }
    Phaser — гибкий заменитель CyclicBarrier, динамическое число участников.
  • 9Что такое Virtual Threads (Java 21, Project Loom)?▶
    Virtual threads — лёгкие потоки управляемые JVM, а не OS. Можно создавать миллионы.
    // Создание
Thread.ofVirtual().start(() -> doWork());
// Через ExecutorService
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
  executor.submit(() -> handleRequest());
}
    Идеально для IO-bound задач: поток блокируется на IO → JVM монтирует его на другой carrier thread → не тратит OS thread. Революция для серверных приложений — простой синхронный код с масштабируемостью как у async!
    Не подходит для CPU-bound задач. Synchronized блоки могут «прикреплять» (pin) виртуальный поток к carrier thread — используй ReentrantLock.

  • 1Расскажи про все области памяти JVM. Что где хранится?▶
    Heap — объекты и массивы. Разделяется потоками. Young Gen (Eden + S0 + S1) + Old Gen. GC работает здесь.
    Stack — фреймы вызовов, локальные переменные примитивов и ссылок. Свой для каждого потока. StackOverflowError = бесконечная рекурсия.
    Metaspace (Java 8+, вместо PermGen) — метаданные классов: методы, поля, байткод, константный пул. Физическая память, не Heap. Растёт динамически. OutOfMemoryError: Metaspace при утечках classloader'ов.
    Code Cache — JIT-скомпилированный нативный код.
    PC Register — адрес текущей JVM-инструкции (у каждого потока свой).
    Native Method Stack — для JNI native методов.
  • 2Young и Old Generation — как работает, зачем поколения?▶
    Гипотеза слабых поколений: большинство объектов умирают молодыми.

    Eden — новые объекты. Быстрое выделение памяти (bump-the-pointer). Minor GC: очищает Eden + один Survivor.
    Survivor (S0, S1) — объекты пережившие GC. Переключаются при каждом Minor GC.
    Old Gen (Tenured) — объекты пережившие N Minor GC (tenuring threshold, ~8-15). Major/Full GC — дорого, stop-the-world.
    // Флаги:
-Xmx512m          // макс heap
-Xms512m          // начальный heap (=Xmx — избежать resize)
-Xmn256m          // размер Young Gen
-XX:SurvivorRatio=8 // Eden:Survivor = 8:1
  • 3Как GC определяет что объект — мусор? GC roots.▶
    Mark and Sweep:
    1. Mark — начиная от GC roots, помечаем все достижимые объекты
    2. Sweep — удаляем непомеченные
    3. Compact (опционально) — сдвигаем оставшиеся для устранения фрагментации

    GC Roots:
    • Локальные переменные и параметры всех активных потоков (стек)
    • static поля классов
    • JNI references
    • Синхронизирующие объекты (мониторы)
    Циклические ссылки (A→B→A) не проблема — они не достижимы из GC roots → оба собираются.
  • 4G1 GC vs ZGC vs Shenandoah — когда что выбрать?▶
    G1 GC (Java 9+ default) — делит heap на регионы (~2048 шт). Предсказуемые паузы (MaxGCPauseMillis). Stop-the-world есть, но короткое. Heap 4-32 GB. Хорош для большинства приложений.

    ZGC (Java 15+ production) — паузы <1ms независимо от размера heap. До терабайт heap. Конкурентное сжатие. Для latency-critical: торговля, игры, real-time.

    Shenandoah — похож на ZGC, паузы ~1-10ms. Меньше накладных расходов на небольших heap.
    // Включить:
-XX:+UseG1GC            // G1 (обычно дефолт)

-XX:+UseShenandoahGC    // Shenandoah

    @ManyToOne, @OneToOne → EAGER (по умолчанию).

    EAGER — коллекция загружается сразу с основной сущностью (JOIN или отдельный запрос).
    LAZY — прокси-объект, коллекция загружается при первом обращении.
    LazyInitializationException — обращение к lazy коллекции вне Hibernate Session.
    Решения: @Transactional, JOIN FETCH в JPQL, @EntityGraph, Spring Data: @Transactional на репозитории.
    Рекомендация: используй LAZY везде + явные JOIN FETCH или @EntityGraph там где нужны данные. Избегай EAGER — он часто ведёт к нежелательным JOIN'ам.
  • 2Проблема N+1 запросов — что это и как решить?▶
    При загрузке N сущностей с lazy коллекцией: 1 запрос для списка + N запросов для каждой коллекции = N+1.
    // ПРОБЛЕМА: при 100 заказах = 101 запрос
List<Order> orders = orderRepo.findAll();
orders.forEach(o -> o.getItems().size()); // N запросов!

// РЕШЕНИЕ 1: JOIN FETCH в JPQL
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.status = :status")
List<Order> findWithItems(@Param("status") Status status);

// РЕШЕНИЕ 2: @EntityGraph
@EntityGraph(attributePaths = {"items", "items.product"})
List<Order> findByStatus(Status status);

// РЕШЕНИЕ 3: @BatchSize (загружает коллекции батчами)
@OneToMany @BatchSize(size = 20)
private List<OrderItem> items;
    Диагностика: spring.jpa.show-sql=true или datasource-proxy для подсчёта запросов. Производительность в N+1 деградирует нелинейно!
  • 3Кеш 1-го и 2-го уровня в Hibernate — как работают?▶
    L1 — Session (Persistence Context): Включён всегда. Хранит сущности в рамках одной Session/транзакции. session.get(User.class, 1L) дважды → 1 SQL запрос.

    L2 — SessionFactory: Разделяется между Sessions. Нужно включить явно (EhCache, Caffeine, Infinispan).
    @Entity
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product { ... }

// application.properties:
spring.jpa.properties.hibernate.cache.use_second_level_cache=true
spring.jpa.properties.hibernate.cache.region.factory_class=...CaffeineCacheRegionFactory
    L2 кеш и высококонкурентные записи = сложность управления инвалидацией. Используй осторожно. Хорошо для: справочники, конфигурации, редко меняющиеся данные.
  • 4Состояния Entity в Hibernate — transient, persistent, detached, removed.▶
    Transient — создан через new, нет в Session и в БД.
    Persistent (Managed) — в Persistence Context. Изменения отслеживаются, auto-flush при commit.
    Detached — был persistent, Session закрыта или evicted. Изменения не отслеживаются. merge() = возврат в managed.
    Removed — помечен на удаление через remove(). Удалится при flush.
    User u = new User("John");  // Transient
session.persist(u);          // → Persistent
u.setName("Jane");           // автоматически UPDATE при flush!
session.close();             // → Detached
u.setName("Bob");            // изменение не отслеживается
session.merge(u);            // → Persistent копия с новыми данными
  • 5Оптимистичная vs Пессимистичная блокировка — когда что?▶
    Оптимистичная (Optimistic) — нет блокировки БД. @Version поле — если версия изменилась при сохранении → OptimisticLockException. Хорошо при редких конфликтах.
    @Entity
public class BankAccount {
  @Version
  private Integer version; // Hibernate инкрементирует автоматически
}
// При конфликте: Spring Data бросает ObjectOptimisticLockingFailureException
    Пессимистичная (Pessimistic) — блокируем строку в БД (SELECT ... FOR UPDATE). Гарантия эксклюзивного доступа. Хорошо при частых конфликтах, критичных операциях (снятие денег).
    @Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Account> findById(Long id);
  • 6Разница между save(), persist(), merge(), saveOrUpdate()?▶
    persist() (JPA) — transient → persistent. Только внутри транзакции. ID не гарантирован до flush.
    save() (Hibernate) — аналог persist но возвращает ID немедленно, работает вне транзакции (insert сразу).
    merge() (JPA) — копирует состояние detached → managed. Возвращает managed копию (не исходный объект!).
    saveOrUpdate() (Hibernate) — save если нет ID, update если есть.
    // ВАЖНО: merge возвращает новый managed объект!
User detached = getDetachedUser();
User managed = session.merge(detached); // managed — это другой объект!
// detached остаётся detached, managed = тот с кем работать дальше
  • 7Как правильно маппить @OneToMany? Cascade и orphanRemoval.▶

public class Order {
  @Id Long id;

  @OneToMany(
    mappedBy = "order",             // поле в OrderItem ссылающееся на Order

  private List<OrderItem> items = new ArrayList<>();

  // Helper методы для поддержания двусторонней связи:
  public void addItem(OrderItem item) {
    items.add(item);
    item.setOrder(this); // ВАЖНО — устанавливаем обе стороны!
  }
}

  • 10Что такое Hibernate dirty checking? Как оптимизировать?▶
    Hibernate сравнивает persistent объекты со снимком (snapshot) при flush — автоматически генерирует UPDATE для изменённых.

  User user = userRepo.findById(id).orElseThrow();
  user.setName("New Name"); // нет явного save()!

}

  • 1Что такое ACID? Объясни каждое свойство.▶
    A — Atomicity (Атомарность): транзакция = атом. Либо всё, либо ничего. Нет частичных изменений.
    C — Consistency (Согласованность): БД переходит из одного корректного состояния в другое. Нарушение constraint → rollback.
    I — Isolation (Изолированность): параллельные транзакции не мешают друг другу (в идеале). Уровень изоляции определяет степень.
    D — Durability (Надёжность): после COMMIT — данные сохранены навсегда, даже при сбое. WAL (Write-Ahead Log) обеспечивает это.
    BASE (Basically Available, Soft state, Eventually consistent) — альтернатива ACID для NoSQL систем, жертвует Consistency ради производительности.
  • 24 уровня изоляции транзакций и их проблемы.▶
    READ UNCOMMITTED — видит незакоммиченные изменения. Dirty read. Самый быстрый, самый опасный.
    READ COMMITTED — видит только закоммиченные. Non-repeatable read (одни и те же данные, разные чтения). Default: PostgreSQL, Oracle, SQL Server.
    REPEATABLE READ — одинаковые данные в рамках транзакции. Phantom read (новые строки появляются). Default: MySQL InnoDB.
    SERIALIZABLE — полная изоляция, как последовательное выполнение. Нет ни одной аномалии. Самый медленный.
    Аномалия         RU   RC   RR   SER
Dirty Read        ✗    ✓    ✓    ✓
Non-Repeatable    ✗    ✗    ✓    ✓
Phantom Read      ✗    ✗    ✗    ✓  (MySQL RR частично защищает)
  • 3Что такое индекс? Как работает B-Tree?▶
    Индекс — отдельная структура данных для быстрого поиска без полного сканирования таблицы.

    B-Tree (сбалансированное дерево) — стандартный тип. O(log n) для поиска, диапазонов, сортировки. Узлы хранят ключи + указатели на строки.
    Hash — O(1) для точного поиска. Не работает для диапазонов, LIKE, ORDER BY.
    GIN/GiST (PostgreSQL) — полнотекстовый поиск, JSONB, массивы.
    Partial index — индекс с условием WHERE.
    CREATE INDEX idx_user_email ON users(email);
CREATE INDEX idx_active_users ON users(email) WHERE active = true; -- partial
CREATE INDEX idx_city_age ON users(city, age); -- составной
    Индекс ускоряет SELECT но замедляет INSERT/UPDATE/DELETE. Не индексируй всё подряд!
  • 4Типы JOIN — INNER, LEFT, RIGHT, FULL, CROSS, SELF.▶
    INNER JOIN — только строки с совпадением в обеих таблицах.
    LEFT JOIN — все из левой + совпадения из правой (NULL если нет). Самый частый.
    RIGHT JOIN — все из правой + совпадения из левой. Редко используют (заменяют LEFT).
    FULL OUTER JOIN — все строки из обеих, NULL где нет совпадения.
    CROSS JOIN — декартово произведение: каждая × каждую. Без ON условия.
    SELF JOIN — таблица сама с собой. Для иерархий (менеджер-сотрудник).
    -- Найти сотрудников и их менеджеров
SELECT e.name, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
  • 5EXPLAIN ANALYZE — как читать план запроса?▶
    EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 AND status = 'ACTIVE';
    На что смотреть:
    • Seq Scan — полное сканирование таблицы. Плохо на больших таблицах.
    • Index Scan — используется индекс. Хорошо.
    • Index Only Scan — все данные из индекса, таблица не читается. Лучшее!
    • Nested Loop / Hash Join / Merge Join — метод JOIN
    • rows — estimated vs actual. Большая разница → устаревшая статистика (ANALYZE)
    • cost — relative cost (не миллисекунды)
    Большое расхождение estimated vs actual rows → ANALYZE table или VACUUM ANALYZE.
  • 6Составной индекс — правило leftmost prefix.▶
    CREATE INDEX idx ON orders(status, user_id, created_at);
    Индекс ИСПОЛЬЗУЕТСЯ:
    WHERE status = ?
    WHERE status = ? AND user_id = ?
    WHERE status = ? AND user_id = ? AND created_at > ?
    ORDER BY status
    ORDER BY status, user_id

    Индекс НЕ используется (нет левого префикса):
    WHERE user_id = ?
    WHERE created_at > ?
    WHERE user_id = ? AND created_at > ?
    Порядок колонок в составном индексе критически важен! Ставь сначала наиболее селективные и наиболее часто используемые в WHERE.
  • 7Оконные функции — ROW_NUMBER, RANK, LAG, LEAD.▶
    Агрегатные вычисления без сворачивания строк — каждая строка сохраняется.
    SELECT
  name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rank,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rank,
  -- (rank пропускает номера при дублях, dense_rank нет)
  AVG(salary)  OVER (PARTITION BY department) AS dept_avg,
  SUM(salary)  OVER (ORDER BY hire_date ROWS UNBOUNDED PRECEDING) AS running_total,
  LAG(salary, 1)  OVER (PARTITION BY department ORDER BY hire_date) AS prev_salary,
  LEAD(salary, 1) OVER (PARTITION BY department ORDER BY hire_date) AS next_salary,
  FIRST_VALUE(name) OVER (PARTITION BY department ORDER BY salary DESC) AS top_earner
FROM employees;
    ROW_NUMBER + подзапрос — классический способ выбрать топ-N из каждой группы!
  • 8Нормализация — 1NF, 2NF, 3NF. Когда денормализовать?▶
    1NF — атомарные значения, нет повторяющихся групп, есть первичный ключ.
    Нарушение: поле "phones" = "111,222,333".
    2NF — 1NF + нет частичной зависимости от составного ключа.
    Нарушение: таблица (order_id, product_id, product_name) — product_name зависит только от product_id.
    3NF — 2NF + нет транзитивных зависимостей.
    Нарушение: (employee_id, dept_id, dept_name) — dept_name зависит от dept_id, а не от employee_id.
    Денормализация уместна для: OLAP/аналитика, read-heavy нагрузка, когда JOIN'ы стали узким местом. Кэшировать агрегаты (total_orders в user), копировать часто используемые поля.
  • 9Что такое Connection Pool? HikariCP настройки.▶
    Connection Pool — набор переиспользуемых соединений с БД. Создание соединения = медленно (TCP + auth). Пул держит готовые соединения.
    HikariCP — самый быстрый пул для Java, дефолт в Spring Boot.
    # application.properties
spring.datasource.hikari.maximum-pool-size=10  # CPU cores × 2 + active disks (для IO)
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.idle-timeout=600000   # 10 мин
spring.datasource.hikari.connection-timeout=30000 # 30 сек
spring.datasource.hikari.max-lifetime=1800000  # 30 мин (меньше timeout сервера!)
    Pool слишком большой = конкуренция в БД (lock contention). Слишком маленький = очередь запросов. Правило: pool = ядра × 2 + число дисков.

  • 1Основные компоненты Kafka — Producer, Consumer, Broker, Topic, Partition, Offset.▶
    Producer — публикует сообщения в топики. Выбирает партицию (по ключу или round-robin).
    Consumer — читает сообщения. Отслеживает offset — свою позицию в партиции.
    Broker — сервер Kafka, хранит данные. Кластер = набор брокеров.
    Topic — логический канал (как таблица). Разделён на партиции.
    Partition — упорядоченный неизменяемый лог сообщений на диске. Единица параллелизма и хранения.
    Offset — уникальный монотонно возрастающий номер сообщения внутри партиции.
    Consumer Group — группа consumers, совместно читающих топик. Каждая партиция — у одного consumer'а в группе.
  • 2Что такое Partition? Зачем нужно несколько партиций?▶
    Партиция — упорядоченный append-only лог. Сообщения внутри партиции строго упорядочены по offset.

    Зачем несколько:
    • Параллелизм — несколько consumers одновременно читают разные партиции
    • Масштабирование — партиции распределяются по брокерам
    • Пропускная способность — параллельная запись/чтение
    Порядок гарантирован только внутри одной партиции! Между партициями — нет. Если нужен порядок для набора сообщений → используй один и тот же ключ → они попадут в одну партицию.
    Число партиций нельзя уменьшить. Увеличение числа партиций → rebalance всех consumer groups.
  • 3Consumer Group и ребалансировка — как работает?▶
    Consumer Group — логическая группа. Каждая партиция назначается ровно одному consumer'у в группе. Разные группы — независимые подписки (каждая читает все сообщения).
    Topic: orders (3 партиции)
Group A: consumer1 читает P0, P1; consumer2 читает P2
Group B: consumer3 читает P0; consumer4 читает P1; consumer5 читает P2
    Rebalance триггеры: consumer присоединился/упал/истёк heartbeat, изменилось число партиций.
    Проблема: долгая обработка → превышение max.poll.interval.ms → consumer считается мёртвым → rebalance → сообщение перечитывается. Решение: увеличить max.poll.interval.ms или уменьшить max.poll.records.
  • 4Что такое ISR (In-Sync Replicas)? Что происходит при падении лидера?▶
    ISR — набор реплик полностью синхронизированных с лидером (не отстающих более чем на replica.lag.time.max.ms).

    Как работает репликация:
    1. Producer отправляет лидеру
    2. Реплики из ISR подтягивают данные
    3. Лидер считает сообщение закоммиченным когда все ISR подтвердили (при acks=all)

    При падении лидера: Kafka Controller выбирает нового лидера из ISR. Только реплики из ISR могут стать лидером (иначе потеря данных).
    Если ISR = 1 (только лидер) + replication.factor=1 → любой сбой = потеря данных. В продакшне: replication.factor=3, min.insync.replicas=2.
  • 5ZooKeeper vs KRaft — зачем переход?▶
    ZooKeeper (legacy) — отдельный кластер для хранения метаданных Kafka, выбора контроллера. Сложность: нужно поддерживать два кластера, сложный ops.

    KRaft (Kafka 3.3+ production ready) — Kafka сама управляет метаданными через Raft консенсус. Брокеры голосуют за контроллера.

    Преимущества KRaft:
    • Проще: нет ZooKeeper = меньше компонентов
    • Быстрее старт: метаданные в памяти
    • Масштабирование до миллионов партиций (ZK ограничивал)
    • Единственная точка правды
    В новых проектах используй KRaft. ZooKeeper mode официально deprecated.
  • 6За счёт чего Kafka такая быстрая? Архитектурные решения.▶
    • Sequential I/O — append-only запись. HDD последовательная = 600 MB/s, случайная = 100 MB/s
    • Page Cache — OS кеширует файлы в RAM. Kafka не держит данные в JVM heap → нет GC overhead
    • Zero Copy — sendfile() syscall: данные из OS cache → NIC без копирования в user space
    • Батчинг — producer накапливает batch → один сетевой вызов, один disk write
    • Сжатие — snappy/lz4/gzip/zstd на уровне батча. lz4 — оптимально CPU/сжатие
    • Partition параллелизм — несколько writer/reader одновременно
  • 7Что такое Log Compaction? Когда использовать?▶
    Log Compaction — для каждого ключа сохраняется только последнее значение. Старые записи с тем же ключом удаляются.
    // Обычный топик (retention.ms=604800000):
key=user1: msg1 → msg2 → msg3  // все 3 хранятся

// Compacted топик (cleanup.policy=compact):
key=user1: msg1 → msg2 → msg3  // только msg3!
    Используй для: key-value state store (последнее состояние объекта), CDC (Change Data Capture), таблицы конфигурации, счётчики. Kafka как вечное хранилище актуального состояния.
    tombstone record — сообщение с value=null означает удаление ключа из compacted топика.
  • 8Как Producer выбирает партицию для сообщения?▶
    Алгоритм:
    1. Если явно указана партиция → используется она
    2. Если есть ключ → partition = hash(key) % numPartitions. Одинаковые ключи → одна партиция → порядок!
    3. Если нет ключа → Sticky Partitioner (Java 2.4+): накапливаем batch в одну партицию, затем меняем. Лучше батчинг чем round-robin.
    // Явная партиция
producer.send(new ProducerRecord<>("topic", 2, key, value));

// По ключу — гарантия порядка для user
producer.send(new ProducerRecord<>("orders", userId.toString(), orderJson));

// Своя стратегия
props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, MyPartitioner.class);
  • 9Kafka vs RabbitMQ — когда что выбрать?▶
    Kafka — выбирай для:
    • Event streaming, event sourcing
    • Высокая пропускная способность (миллионы сообщений/сек)
    • Хранение истории сообщений, replay
    • Multiple consumers с разными позициями
    • Log aggregation, CDC, аналитика
    RabbitMQ — выбирай для:
    • Сложная маршрутизация (exchanges: direct, topic, fanout, headers)
    • Priority queues
    • Задачи (task queue) — exactly-once потребление
    • Меньше инфраструктурных требований
    • Когда нужен AMQP
    Kafka — append-only лог. RabbitMQ — традиционный брокер с удалением сообщений после получения.

  • 13 гарантии доставки — at-most-once, at-least-once, exactly-once.▶
    At-most-once — сообщение доставлено максимум раз. Возможна потеря при сбое. acks=0, retries=0. Для метрик, логов где потеря допустима.

    At-least-once — доставлено минимум раз. Дубли возможны. acks=all, retries>0, manual commit. Consumer должен быть идемпотентен.

    Exactly-once (EOS) — ровно 1 раз. Без потерь и дублей. enable.idempotence=true + транзакции Kafka. Самое надёжное, есть overhead.
    На практике at-least-once + идемпотентный consumer — оптимальный выбор для большинства систем.
  • 2acks=0 vs acks=1 vs acks=all — детально.▶
    acks=0 — не ждать подтверждений. Самый быстрый. Данные могут потеряться при любом сбое.

    acks=1 — лидер подтверждает запись в свой лог. Данные потеряются если лидер упадёт до репликации. Компромисс.

    acks=all (-1) — все реплики из ISR подтверждают. Максимальная надёжность. Медленнее. Комбинируй с min.insync.replicas=2.
    props.put(ProducerConfig.ACKS_CONFIG, "all");
props.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
  • 3Идемпотентный Producer — как настроить и как работает?▶
    Producer получает уникальный PID (Producer ID). Каждое сообщение получает sequence number. Брокер дедуплицирует по (PID, partition, seqNum).
    props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
// Автоматически:
// acks = all
// retries = Integer.MAX_VALUE
// max.in.flight.requests.per.connection <= 5
    Идемпотентность защищает от дублей при ретраях (сеть упала, ответ не дошёл). НО: только в пределах одной сессии продюсера. При перезапуске продюсера — новый PID, защиты нет → нужна транзакционность или идемпотентность consumer'а.
  • 4Ручной vs автоматический commit offset — разница?▶
    auto.commit=true — offset коммитится каждые auto.commit.interval.ms (5 сек). Если consumer упал после коммита но до обработки → потеря. Простой, не даёт контроля.

    Ручной commit — полный контроль:
    while (true) {
  ConsumerRecords<K, V> records = consumer.poll(Duration.ofMillis(100));
  for (ConsumerRecord<K, V> record : records) {
    process(record);
  }
  // commitSync() — надёжно, блокирует
  consumer.commitSync();
  // commitAsync() — быстро, при ошибке может пропустить
}
    Коммить после обработки, не до! Иначе сообщение считается обработанным, а обработка не произошла.
